3. Supabase — Integração com Banco na Nuvem NOVO

3.1 O que é o Supabase?

Supabase é uma plataforma Backend-as-a-Service que fornece um banco de dados PostgreSQL gerenciado na nuvem, gratuitamente no plano free. Em vez de depender de um PostgreSQL instalado localmente, o banco fica em servidores da AWS e está sempre acessível.

Projeto VORTEX ERP no Supabase:
Project ID: inktgatsnofilqdkqhbz
Host: db.inktgatsnofilqdkqhbz.supabase.co
Região: US East (Virginia)

3.2 O que mudou no código

Antes (local)Depois (Supabase)
postgresql://postgres:1234@localhost:5432/vortex_erppostgresql://postgres:%40...@db.inktgatsnofilqdkqhbz.supabase.co:5432/postgres
PostgreSQL instalado no PCPostgreSQL na nuvem (AWS)
Dados só existem no seu computadorDados acessíveis de qualquer lugar
Precisa do PostgreSQL rodando localmenteSem necessidade de instalação local

Apenas o arquivo .env mudou. Todo o código do backend, as rotas, o Prisma e o frontend continuam exatamente iguais — porque o Prisma abstrai o banco.

7. Backend — Node.js + Express + TypeScript

7.1 Arquivo src/index.ts — Servidor principal

import 'dotenv/config';       // carrega .env primeiro
import express from 'express';
import cors from 'cors';
import authRoutes      from './routes/auth';
import usersRoutes     from './routes/users';
import clientesRoutes  from './routes/clientes';
import estoqueRoutes   from './routes/estoque';
import financeiroRoutes from './routes/financeiro';
import vendasRoutes    from './routes/vendas';

const app = express();
app.use(cors({ origin: (origin, cb) => {
  const ok = !origin || /^https?:\/\/(localhost|127\.0\.0\.1)(:\d+)?$/.test(origin);
  cb(null, ok);
}, credentials: true }));
app.use(express.json());

app.use('/auth',          authRoutes);
app.use('/api/users',     usersRoutes);
app.use('/api/clientes',  clientesRoutes);
app.use('/api/estoque',   estoqueRoutes);
app.use('/api/financeiro', financeiroRoutes);
app.use('/api/vendas',    vendasRoutes);

app.listen(3000);

7.2 Middleware de autenticação: src/middleware/auth.ts

Executado antes de qualquer rota protegida. Valida o JWT no header Authorization: Bearer <token>. Se inválido → retorna 401. Se válido → coloca req.user = {id, email, role} e continua.

8.2 Como configurar o Gmail SMTP

  1. Acesse myaccount.google.com
  2. Vá em Segurança → Verificação em duas etapas (deve estar ativada)
  3. Pesquise "Senhas de app" na barra de pesquisa da conta Google
  4. Crie uma senha de app com nome "Vortex ERP"
  5. Copie os 16 caracteres gerados (formato: xxxx xxxx xxxx xxxx)
  6. No arquivo .env: SMTP_PASS=xxxx xxxx xxxx xxxx
⚠️ Atenção: Use sua senha de App do Google — não a senha normal da conta. A senha normal não funciona com SMTP.

// Lê o token JWT do localStorage (salvo no login)

  post: <T>(path: string, body: unknown) =>

  // get, put, patch, del — mesmo padrão

11.2 Como funciona o bcrypt

// Cadastro — nunca salva senha em texto
const hash = await bcrypt.hash('MinhaSenh@123', 12);
// Salva no Supabase: "$2b$12$..."

// Login — compara sem descriptografar
const ok = await bcrypt.compare('MinhaSenh@123', hash); // true

11.3 Estrutura do JWT

// Payload armazenado no token (não criptografado, apenas assinado)

12. Fluxos Completos ATUALIZADO v2.0

12.1 Cadastro de Novo Usuário (com verificação de e-mail)

1

Usuário preenche o formulário de cadastro

Nome, e-mail e senha. Frontend envia POST /auth/register.

2

Backend verifica se o e-mail já existe

Se existir → retorna 409 email_em_uso. Se não → continua.

3

Cria o usuário com emailVerified: false

Senha é hasheada com bcrypt (12 rounds). Gera código de 6 dígitos com validade de 15 minutos.

4

Envia e-mail de verificação

Via Nodemailer + Gmail SMTP. O e-mail contém o código de 6 dígitos. Retorna: {"success":true, "message":"Verifique seu e-mail"}.

5

Usuário acessa /verificar-email

Digita seu e-mail e o código recebido. Frontend envia POST /auth/verify-email.

6

Backend valida o código

Verifica: código correto? Dentro do prazo? Se ok → atualiza emailVerified: true no Supabase. Limpa o código.

7

Usuário pode agora fazer login

Acessa /login com e-mail e senha. O login verifica que emailVerified = true e gera o JWT.

12.2 Login

1

POST /auth/login com {email, password}

Backend busca o usuário no Supabase pelo e-mail.

2

Verifica bloqueios

Conta bloqueada por tentativas? → 423. Status bloqueado pelo admin? → 423.

3

Compara senha com bcrypt

Errada? → incrementa loginAttempts. Após 5 falhas → bloqueia 15 minutos.

4

Verifica emailVerified

Se false → retorna 403 email_nao_verificado. Frontend orienta o usuário a verificar o e-mail.

5

Sucesso — gera JWT de 8 horas

Zera as tentativas. Retorna {token, user}. Frontend salva no localStorage e redireciona ao Dashboard.

12.3 Recuperação de Senha

1

POST /auth/forgot-password com {email}

Sempre retorna sucesso (evita enumerar e-mails). Gera código de 6 dígitos, salva em PasswordResetCode, envia e-mail.

2

POST /auth/validate-code com {email, code}

Valida: correto? Não usado? Dentro do prazo? Se ok → retorna resetToken JWT (15 min, purpose: 'password_reset').

3

POST /auth/reset-password com {resetToken, newPassword}

Verifica o resetToken. Gera novo hash bcrypt. Atualiza a senha no Supabase. Envia e-mail de confirmação "sua senha foi alterada".

12.4 Como o Admin desbloqueia uma conta

Após 5 tentativas erradas, a conta fica bloqueada por 15 minutos. O admin pode desbloquear manualmente:

  1. Acessar a página Usuários (somente admin)
  2. Clicar em "Desbloquear" no usuário → chama PATCH /api/users/:id/status
  3. Backend zera loginAttempts = 0 e lockedUntil = null no Supabase
Alternativa direta no Supabase: Painel do Supabase → Table Editor → tabela User → edite os campos loginAttempts (0) e lockedUntil (null) do usuário.

15. Segurança

✅ O que está protegido

ProteçãoImplementação
Senhas hasheadasbcrypt 12 rounds — impossível reverter
JWT assinadoQualquer alteração no token o invalida
Bloqueio por tentativas5 falhas → bloqueio 15 min no Supabase
Verificação de e-mailNovo usuário não faz login sem confirmar o e-mail v2.0
Enumeração de e-mailforgot-password sempre retorna sucesso independente do e-mail existir
RBAC no backendServidor verifica o role em cada rota — frontend não é suficiente
Token de propósitoresetToken tem campo purpose:'password_reset' — não vira token de login
Banco na nuvemSupabase gerencia backups, patches de segurança e HA automaticamente

⚠️ Vulnerabilidades a corrigir

🔴 CRÍTICO — JWT_SECRET fraco: Gere com:
node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"
🟡 Sem HTTPS em produção: Instale certificado SSL (Let's Encrypt) + Nginx. Sem HTTPS, o JWT trafega exposto.
🟡 Sem rate limiting: Instale express-rate-limit para limitar tentativas por IP.
🟡 CORS aberto para localhost: Em produção, altere o CORS para aceitar apenas o domínio real:
app.use(cors({ origin: 'https://seu-dominio.com' }))

16. Módulos do ERP

Clientes

CRUD completo com busca em tempo real por nome, e-mail, telefone e CPF/CNPJ. A busca não faz chamada ao servidor — filtra os dados já carregados no React com .filter().

Estoque

Controle de produtos com badges de status por quantidade: Sem estoque = 0, Crítico ≤5, Baixo ≤15, Normal >15. O ajuste de quantidade usa PATCH /api/estoque/:id/quantidade com campo delta (positivo ou negativo).

Vendas — Kanban

Quadro com 4 colunas: Aberto → Em Andamento → Concluído / Cancelado. Mover um card: PATCH /api/vendas/:id/status. Clicar em "Ver" abre modal com detalhes dos itens.

Financeiro

Receitas e despesas com resumo de saldo no topo. Filtro por tipo sem chamada extra. Botão "✓ Pagar" alterna o status de pagamento.

2

Criar o projeto backend

mkdir VORTEX_ERP && cd VORTEX_ERP && npm init -y

3

Instalar dependências

npm install express @prisma/client bcrypt jsonwebtoken nodemailer cors dotenv
npm install --save-dev typescript ts-node prisma @types/express @types/node @types/bcrypt @types/jsonwebtoken @types/nodemailer @types/cors

4

Configurar Prisma e .env

npx prisma init → edite o .env com a DATABASE_URL do Supabase (encode o @ da senha como %40).

5

Criar schema.prisma

Copie todos os modelos do capítulo 6. Inclua os campos emailVerified, emailVerificationCode, emailVerificationExpiry no User.

6

Rodar migration no Supabase

npx prisma migrate dev --name inicio → cria todas as tabelas diretamente no Supabase.

7

Criar código do backend

Na ordem: src/prisma.ts → src/mailer.ts (3 funções) → src/middleware/auth.ts → src/routes/*.ts → src/index.ts

8

Seed e teste

npx prisma db seed → npm run dev → teste com curl ou Postman.

9

Criar frontend React

npm create vite@latest frontend-react -- --template react-ts
cd frontend-react && npm install && npm install react-router-dom

10

Criar páginas React

api/client.ts → contexts/AuthContext.tsx → components/ → pages/ (incluindo VerificarEmail.tsx) → App.tsx

11

Iniciar tudo

Terminal 1: cd VORTEX_ERP && npm run dev
Terminal 2: cd frontend-react && npm run dev
Acesse: http://localhost:5173
